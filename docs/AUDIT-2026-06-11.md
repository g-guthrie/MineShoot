# Codebase Audit â€” 2026-06-11

Full-stack audit of ShooterFinal as the foundation for a browser multiplayer FPS.
Six review passes: game server (zombies-game), client network/core, client rendering/UI,
legacy-stack assessment, multiplayer security/trust boundary, and engineering infrastructure.

**Verdict: the foundation is worth keeping.** The Hytopia stack (zombies-game +
hytopia-client) is architecturally sound â€” genuinely server-authoritative combat/economy,
a well-engineered worker-based rendering pipeline, and a small (~75-line) well-isolated
fork delta. The problems are concentrated in four areas: (1) lifecycle hygiene on the
server (timers/state outliving despawn), (2) error-handling on the client (fire-and-forget
async paths that silently kill rendering or the HUD), (3) the local-dev networking changes
overreaching (every IPv4 treated as "local"), and (4) repo/process debt (red CI, a live
legacy game nobody decided the fate of, a license-compliance gap, uncommitted dev
entrypoints).

## Server (zombies-game) â€” gameplay bugs

### High
- **`EnemyEntity.ts:102` â€” money reward uses the wrong variable.** Pays
  `(this.damage / maxHealth) * reward` where `this.damage` is the enemy's *contact-damage*
  stat, not the bullet damage parameter. Weak fast guns systematically out-earn; total
  payout per kill is decoupled from `reward`; shotgun pays per pellet (up to 7Ã—/trigger);
  the "half reward on kill" comment at :106 is unimplemented. Found independently by two
  reviewers. Fix: use the `damage` param clamped to remaining health; consider a kill bonus.
- **`GamePlayerEntity.ts:188-212` â€” revive timer survives despawn.** `progressRevive`
  re-arms recursively with no `isSpawned` guard and no cleanup on despawn/disconnect.
  Despawned entities report position `{0,0,0}`, so distance=0 passes the check and a dead
  entity gets "healed" while stale HUD updates clobber the respawned player's UI. Fix:
  clear on `EntityEvent.DESPAWN`; guard callback on `this.isSpawned && byPlayer.isSpawned
  && this.downed`. Related: mashing E resets the 1s timer each press, so revive never
  progresses for a button-mashing reviver (`if (this._reviveInterval) return` fixes it).
- **`GameManager.ts:154,169-173` â€” 250ms double-spawn window in `endGame()`.**
  `isStarted=false` is set synchronously but respawn happens in a 250ms setTimeout; a
  player joining in the window gets two entities, both wired to their input (double fire,
  double earn). Fix: bail in `spawnPlayerEntity` if the player already has an entity.
- **`WeaponCrateEntity.ts:123-134` â€” crate roll is crate-scoped and never expires.**
  Anyone can claim a roll someone else paid for, and unclaimed rolls persist across
  `endGame()` (crates are never reset), giving free elite weapons next game. Fix: bind
  roll to purchaser with a timeout; reset crates in `endGame()`.
- **`index.ts:3-8` â€” global uncaughtException swallower masks state corruption.** Keeps
  the process alive after any throw, including mid-physics-tick; exists to paper over the
  despawn races above. After fixing them, replace log-and-continue with log-and-restart.

### Medium
- Disconnect+reconnect mid-game relegates the player to spectator with no HUD until next
  round (`index.ts:42-48`); hytopia-game's `RECONNECTED_WORLD` handler shows the pattern
  to port. Track recent players by `player.id` and respawn on rejoin.
- `checkEndGame()` only fires from `_setDowned` and the spawn loop â€” when the last
  standing player disconnects, game-over lags up to ~14s; call it from `LEFT_WORLD`.
- Start countdown freezes (doesn't reset) when the lobby empties; the next joiner can get
  ambushed by a 3-second start. Interval also runs forever and spams chat every second.
- Reload completing on a swapped-out gun overwrites the new gun's ammo HUD
  (`GunEntity.ts:159-167`; `despawn()` does not clear `_parent`, so the `isEquipped`
  guard passes). Guard on `this.isSpawned`.
- Money is fractional and never rounded (HUD floats; `spendMoney` comparisons can fail at
  displayed-equal values). Round to integers.
- AutoShotgun spawns with 6 ammo in a 4-round magazine (`AutoShotgunEntity.ts:9,13`).
- `PistolEntity.shoot()` hardcodes a duplicate `shoot_gun_right` animation restart that's
  wrong for the two-handed guns inheriting it (`PistolEntity.ts:47`).
- Wave mechanics: spawn loop starts before wave loop (first zombie has wave-0 stats), and
  the 10s inter-wave delay attaches to one arbitrary spawn instead of pausing the loop.

### Low (selected)
- `endGame` respawns via `getConnectedPlayers()` (all worlds) vs. by-world everywhere else.
- Two simultaneous Rippers share one boss-bar UI channel.
- Enemies chase downed (untargetable) players up to 5s; contact damage fires only on
  collision-start (sustained contact â‰ˆ no damage).
- Barrier orientation inferred via float equality `rotation?.w === 1`.
- Per-tick allocation in `_getTargetDistance`; A* pathfinding bursts synchronized across
  enemies spawned together (jitter the accumulators).
- Gun classes are ~90% data: a single `WEAPONS` registry + `semiAuto` flag would delete
  ~6 files and the crate's class switch. Same for Zombie/Ripper.
- `net.Server.prototype.listen` monkeypatch in `index.ts` is global and arity-fragile.

## Client (hytopia-client) â€” network & core

### High
- **`Servers.ts:13-23` (uncommitted) â€” `isLocalServer` matches every IPv4 address**, not
  just loopback, and now drives protocol selection everywhere: a TLS-fronted production
  server on a bare IP gets forced to plaintext `ws://` (breaks outright, plus
  mixed-content blocking from an https-served client); a LAN playtest at `192.168.x.x`
  fails Chromium's Private Network Access because the code declares
  `targetAddressSpace: 'loopback'` for private-space endpoints. The generalization
  defeats the exact use cases it was built for. Fix: classify only loopback
  (`127.0.0.0/8`, `localhost`, `[::1]`) as local; better, make protocol/endpoint explicit
  config instead of inferring from hostname shape.
- **`NetworkManager.ts:578-595` â€” reconnect ignores its own health check and has no
  backoff.** Every `onclose` (including failed handshakes) triggers a full page reload â€”
  a failed WS upgrade produces an infinite reload loop. `_connectWebSocket()`'s promise
  never settles on failure; `connect()`'s sync/heartbeat intervals are never cleared
  after a server `kill` packet. Fix: use `serverHealthy` to gate reload vs. error UI,
  capped backoff, settle the promise on close, clear intervals in `_killConnection()`.
- **`scripts/local-hytopia-proxy.mjs` â€” crashes the whole process on routine socket
  errors.** Neither piped socket after a WS upgrade has an `'error'` listener; one
  ECONNRESET (tab close, sleep/wake, server restart) kills the proxy for every connected
  client. Also hangs the client forever if the upstream answers the upgrade with a non-101
  response. Fix: error/close handlers destroying the peer socket; handle `'response'`.

### Medium
- `EntityManager.ts:316-325` â€” missing `return` after entity removal: released entities
  keep processing the rest of the packet; a batched `removed+modelUri` resurrects an
  untracked mesh (permanent leak).
- One throw in `_updateEntity` aborts the rest of the entities batch (silent desync).
- `Assets.urlExists` caches transient failures forever (one flaky HEAD at join â†’ blocks
  never render, no recovery) and feeds fire-and-forget `_buildGLTFModel()` calls.
- `Servers.ts` compat-redirect can ship players off the fork to HYTOPIA's hosted client;
  delete it â€” the fork is the only client.
- `BridgeManager` accepts `postMessage` from any origin and broadcasts chat/player data
  with `'*'` target.
- `_validateServerVersionCompat` crashes on version-less server JSON (fallback ordered
  after `.includes('DEV')`).
- WebTransport stream-failure path reloads the page mid-fallback instead of falling back.

### Fork posture
Live delta vs upstream is ~75 lines across 5 files â€” sustainable, but there's no recorded
upstream commit/baseline. Record it in a VENDOR.md, keep local changes as a patch series,
and pin `hytopia` (server) + `@hytopia.com/server-protocol` (client) in lockstep
(zombies-game currently declares `"hytopia": "*"`).

## Client â€” rendering & UI

### High
- **`ChunkWorker.ts:285-351` â€” message pump permanently stalls if any handler throws**

  `_dequeue` in try/catch/finally, post errors to main thread.
- **`Game.ts:126` â€” block-atlas init is fire-and-forget** and combines with the

  `Renderer._loadSkybox`.
- **`BlockMaterialManager.ts:337-348` â€” `MeshBlockMaterial.clone()` produces a corrupt
  material** (`_game === undefined`, lost `defines`); doesn't crash today only because
  the one caller immediately overwrites `onBeforeCompile`. Override clone/copy like
  `EmissiveMeshBasicMaterial` does.

### Medium

  half-batch camera movement.

- Water animation speed is frame-rate dependent (`+= 0.0075` per frame).

## Security / trust boundary

**Result: genuinely server-authoritative.** Fire rate, ammo, reload, raycast/range/wall
occlusion, damage, money, interact distance, revive timing, and movement (SDK
input-only model) are all server-enforced â€” the classic cheats (rapid fire, infinite
ammo, wallshots, range hacks, free money, teleport, speedhack) are not exploitable.

Remaining surface:
- **Aimbot (inherent, medium):** shoot direction is the client-reported camera facing,
  unvalidated (`GunEntity.ts:121`). Origin and range are server-anchored, so this is the
  only real cheat vector left. Mitigation: server-side angular-velocity/snap heuristics.
- **Chat flooding (medium):** no rate limit in SDK or game code; broadcast amplification.
  Add a per-player token bucket on `ChatEvent.PLAYER_MESSAGE`.
- **Countdown griefing (low):** join/leave spam stalls the pre-game countdown
  indefinitely; no spawn/connection caps in game code.
- **Proxy (informational):** dev-only and correctly loopback-bound; must never be exposed
  (cert-blind, TLS-stripping, unauthenticated relay).
- **Secrets: clean.** No committed credentials; one hardcoded Cloudflare account ID
  (identifier, not a secret) in `deploy-status.sh`.
